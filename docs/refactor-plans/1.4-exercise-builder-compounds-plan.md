## Current Component Analysis

### State Inventory

| State Variable | Used By |
|---|---|
| `loading` | Toolbar save buttons |
| `error` | Error banner |
| `saveSuccess` | Toolbar save button indicator |

Derived: `isAudioType`, `isDetailsComplete`, `hasAIConfig`, `steps[]`

### Key Observations

1. **Simpler than lesson-editor** -- no preview panel, no mobile footer duplication, no content mode toggle. Only 726 lines vs 837.
2. **Steps are dynamic** -- the steps array conditionally includes either `ai-config` or `audio-config` based on `exercise_type`. This is an important detail for context.
3. **No button duplication** -- save/publish buttons appear only once (in toolbar), unlike lesson-editor which had desktop + mobile copies.
4. **Error banner is tiny** (~14 lines) -- keep inline in orchestrator.
5. **Three substantial step panels** -- Details (~170 lines), AI Config (~110 lines), Audio Config (~150 lines). All worth extracting.

### What NOT to Split

- **Error Banner** (~14 lines): Too small, tightly coupled to `error`/`setError`.
- **Constants** (`DIFFICULTY_COLORS`, `TYPE_ICONS`): Move to context file alongside types, or keep in details step since that's the only consumer.

## Shared Context Interface

```typescript
// exercise-builder-context.tsx

export interface ExerciseBuilderProps {
  courseId: number
  lessonId?: number
  initialData?: any
}

export type ExerciseStep = 'details' | 'ai-config' | 'audio-config'

export interface ExerciseFormData {
  title: string
  description: string
  instructions: string
  exercise_type: string
  difficulty_level: string
  time_limit: number
  system_prompt: string
  status: string
  // audio/video config
  topic_prompt: string
  min_duration_seconds: number
  max_duration_seconds: number
  passing_score: number
  max_daily_attempts: number
  rubric_filler_words: boolean
  rubric_pace: boolean
  rubric_structure: boolean
  rubric_confidence: boolean
}

export interface StepDefinition {
  key: ExerciseStep
  label: string
  icon: React.ReactNode
  complete: boolean
}

export interface ExerciseBuilderContextValue {
  // State
  formData: ExerciseFormData
  loading: boolean
  error: string | null
  saveSuccess: boolean
  activeStep: ExerciseStep

  // Props pass-through
  initialData: ExerciseBuilderProps['initialData']
  courseId: number

  // Derived
  isAudioType: boolean
  isDetailsComplete: boolean
  hasAIConfig: boolean
  steps: StepDefinition[]

  // Actions
  updateField: <K extends keyof ExerciseFormData>(key: K, value: ExerciseFormData[K]) => void
  setFormData: React.Dispatch<React.SetStateAction<ExerciseFormData>>
  handleSave: (publish: boolean) => Promise<void>
  setActiveStep: (step: ExerciseStep) => void
  setError: (error: string | null) => void
}

const ExerciseBuilderContext = createContext<ExerciseBuilderContextValue | null>(null)

export function useExerciseBuilder() {
  const ctx = use(ExerciseBuilderContext)
  if (!ctx) throw new Error('useExerciseBuilder must be used within ExerciseBuilderProvider')
  return ctx
}
```

Uses React 19 `use()` for context consumption (same pattern as lesson-editor).

## Import Path Preservation

Dynamic imports in the page files use:
```tsx

```

The barrel `index.tsx` re-export preserves this exact path. The old `exercise-builder.tsx` file becomes a directory `exercise-builder/` with `index.tsx`. No changes needed in:
- `app/[locale]/dashboard/teacher/courses/[courseId]/exercises/new/page.tsx`

## Key Decisions

1. **No separate actions component** -- Unlike lesson-editor which had duplicate save buttons (desktop + mobile), exercise-builder has save buttons only in the toolbar. No need for a shared `ExerciseBuilderActions` component.

2. **`DIFFICULTY_COLORS` moves to details step** -- It's only used there. `TYPE_ICONS` is unused (the select items use inline emoji strings), so it can be removed.

3. **Dynamic steps stay in context** -- The conditional `ai-config` vs `audio-config` step logic belongs in the provider since it depends on `formData.exercise_type` and is consumed by the toolbar.

4. **`isAudioType` exposed via context** -- The orchestrator needs it to decide which step panel to show (though currently the step buttons only show one or the other, the `activeStep` value itself handles routing). Exposing it allows the orchestrator to remain simple.
