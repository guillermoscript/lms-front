# 0.6 — Shared Server Action Patterns

## Status: COMPLETE

## Problem

The codebase has ~30 server action files with inconsistent patterns:

| Concern | Current State |
|---------|--------------|
| **Result type** | `ActionResult<T>` exists in `lib/supabase/admin.ts` but only ~40% of actions use it. Others throw errors or return ad-hoc `{ success: true }` objects. |
| **Auth validation** | Every action duplicates 3-5 lines: `getUser()`, `getUserRole()`, `getCurrentTenantId()`. `community.ts` has a local `getAuthenticatedUser()` helper but it is not shared. |
| **Role checks** | Some use `verifyAdminAccess()` from `lib/supabase/admin.ts`, others inline the check. |
| **Error handling** | **Mixed**: teacher/courses.ts and payment-requests.ts **throw** errors. products.ts, community.ts, lesson-resources.ts **return** `{ success: false, error }`. Callers must know which convention each action uses. |
| **Ownership verification** | Duplicated per file (e.g. `verifyLessonOwnership` in lesson-resources.ts, inline course ownership checks in teacher/courses.ts). |

## Analysis of Existing Actions

Files analyzed in detail:
- `app/actions/teacher/courses.ts` — throws errors, no ActionResult
- `app/actions/teacher/lesson-resources.ts` — returns ActionResult, has local verifyLessonOwnership
- `app/actions/payment-requests.ts` — mixed (some throw, some return objects)
- `app/actions/admin/products.ts` — returns ActionResult, uses verifyAdminAccess
- `app/actions/community.ts` — returns ActionResult, has local getAuthenticatedUser
- `app/actions/exam-grading.ts` — returns custom result type, no ActionResult

### What is ALREADY consistent
- All files use `'use server'` directive
- All files import from `@/lib/supabase/server` and `@/lib/supabase/tenant`
- All files call `supabase.auth.getUser()` for authentication (not `getSession()`)
- All files filter by `tenant_id`
- All files use `revalidatePath()` after mutations

### What is INCONSISTENT
- Error propagation: throw vs return `{ success: false }`
- Auth context gathering: 3-5 duplicated lines per action
- Role verification: inline vs `verifyAdminAccess()`
- Course/lesson ownership checks: duplicated logic

## Solution: Shared Utility (`lib/actions/utils.ts`)

Created `lib/actions/utils.ts` with these exports:

### 1. `AuthContext` type
```typescript
interface AuthContext {
  user: User
  userId: string
  tenantId: string
  role: 'student' | 'teacher' | 'admin'
  supabase: SupabaseClient
}
```

### 2. Auth helpers (replace 3-5 duplicated lines per action)
```typescript
authenticateUser()        // Returns AuthContext, throws if not authed
requireTeacherOrAdmin()   // Returns AuthContext, throws if wrong role
requireAdmin()            // Returns AuthContext, throws if not admin
```

### 3. `verifyCourseOwnership(ctx, courseId)`
Shared check that the course belongs to the tenant and the user (teacher) or tenant (admin). Phase 1 builders all operate on courses.

### 4. `actionHandler<T>(fn)` — error boundary wrapper
```typescript
export async function myAction(id: number) {
  return actionHandler(async () => {
    const ctx = await requireTeacherOrAdmin()
    // ... mutations ...
    return { data: 'here' }
  })
}
// Always returns { success: true, data: T } | { success: false, error: string }
```

This normalizes all actions to **never throw** to callers, while internal code can throw freely.

## Convention for Phase 1 Actions

All new server actions for lesson-editor, exercise-builder, and exam-builder MUST follow:

### Template
```typescript
'use server'

import { actionHandler, requireTeacherOrAdmin, verifyCourseOwnership } from '@/lib/actions/utils'
import type { ActionResult } from '@/lib/actions/utils'
import { createAdminClient } from '@/lib/supabase/admin'
import { revalidatePath } from 'next/cache'

export async function updateLesson(
  courseId: number,
  lessonId: number,
  data: LessonFormData
): Promise<ActionResult<{ id: number }>> {
  return actionHandler(async () => {
    // 1. Auth + role check (one line)
    const ctx = await requireTeacherOrAdmin()

    // 2. Ownership check (one line)
    await verifyCourseOwnership(ctx, courseId)

    // 3. Validate input
    if (!data.title?.trim()) throw new Error('Title is required')

    // 4. Perform mutation (use ctx.supabase or createAdminClient)
    const { data: lesson, error } = await ctx.supabase
      .from('lessons')
      .update({ title: data.title.trim() })
      .eq('id', lessonId)
      .eq('tenant_id', ctx.tenantId)
      .select('id')
      .single()

    if (error) throw error

    // 5. Revalidate
    revalidatePath(`/dashboard/teacher/courses/${courseId}`)

    // 6. Return data (actionHandler wraps in { success: true, data })
    return lesson
  })
}
```

### Rules
1. **Always return `ActionResult<T>`** — use `actionHandler()` wrapper.
2. **Never throw to callers** — `actionHandler` catches and formats errors.
3. **Auth in one line** — use `authenticateUser()`, `requireTeacherOrAdmin()`, or `requireAdmin()`.
4. **Course ownership in one line** — use `verifyCourseOwnership(ctx, courseId)`.
5. **Validate inputs** before mutations — throw on invalid input (caught by `actionHandler`).
6. **Use `ctx.supabase`** for RLS-scoped queries, `createAdminClient()` for service-role ops.
7. **Always filter by `ctx.tenantId`** in queries.
8. **Call `revalidatePath()`** after successful mutations.

### Client-side consumption
```typescript
const result = await updateLesson(courseId, lessonId, formData)
if (!result.success) {
  toast.error(result.error)
  return
}
// result.data is typed as { id: number }
```

## What This Does NOT Do

- Does NOT refactor existing actions — they keep working as-is.
- Does NOT add Zod validation — input validation stays inline for now.
- Does NOT change `ActionResult` definition — it stays in `lib/supabase/admin.ts` and is re-exported from `lib/actions/utils.ts`.
- Does NOT add `actionHandler` to existing actions — only Phase 1 new actions will use it.

## Relationship to Phase 1

Phase 1 subtasks (lesson-editor, exercise-builder, exam-builder) will:
1. Extract mutations from client components into new action files under `app/actions/teacher/`
2. Use the patterns from `lib/actions/utils.ts`
3. Have consistent error handling that client components can rely on
