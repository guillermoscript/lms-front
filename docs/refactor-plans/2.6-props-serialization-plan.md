# 2.6 Props Serialization Plan — Minimize Data Passed to Client Components

## Analysis Summary

Reviewed 4 student pages for over-serialization at RSC boundaries.

### Pages Analyzed

#### 1. `app/[locale]/dashboard/student/page.tsx` — FIXED

**Issues found:**
- `exam_submissions` used `select('*')` but `RecentActivity` client component only needs `submission_id`, `score`, `submission_date`, `exam_id` (4 fields out of ~10+).
- `lesson_completions` used `select('*')` but only `.length` is used (just needs a count).
- `upcomingExams` used `select('*, course:courses(title)')` — all exam columns serialized but client only uses `exam_id`, `title`, `exam_date`, `course.title`.
- `enrollments` outer select used `*` — all enrollment columns fetched, but only the nested `course` data is used (enrollment columns discarded by the mapping on line 77-93).

**Applied fix:**
- `exam_submissions`: `select('*')` -> `select('submission_id, exam_id, score, submission_date')`
- `lesson_completions`: `select('*')` -> `select('lesson_id')`
- `upcomingExams`: `select('*')` -> `select('exam_id, title, exam_date, course:courses(title)')`
- `enrollments`: `select('*')` -> `select('enrollment_id, course:courses(...))'`
- Note: `upcomingExams` required `as any[]` cast because Supabase type generation infers `course` as array type when using explicit columns (vs `*` which correctly infers single-object for FK relations).

#### 2. `app/[locale]/dashboard/student/courses/[courseId]/lessons/[lessonId]/page.tsx` — FIXED

**Issues found:**
- Main query used `select('*, courses(*), ...')` which fetched ALL columns from `lessons` and `courses` tables.
- From `lessons`, only these fields are used: `id`, `title`, `sequence`, `content`, `video_url`, `embed_code`.
- From `courses`, only: `title`, `require_sequential_completion`.

**Applied fix:**
- `lessons` select: `*` -> `id, title, sequence, content, video_url, embed_code, course_id`
- `courses(*)` -> `courses(title, require_sequential_completion)` (with type cast to handle Supabase FK inference)
- `lessons_ai_tasks(*)` -> `lessons_ai_tasks(task_instructions)` (only field used)
- `lessons_ai_task_messages(*)` -> `lessons_ai_task_messages(id, message, sender, tool_invocations, created_at)` (only fields used)
- Kept `lesson_comments(*, profiles(*), comment_reactions(*))` as-is — the data is transformed server-side into a tree structure before crossing the boundary, so changing the select would break the existing type cast.
