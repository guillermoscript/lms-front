# 1.6 Split exam-builder.tsx into Compound Components

## Source File
`components/teacher/exam-builder.tsx` (617 lines)

## Current Structure

The file is a single monolithic `ExamBuilder` component with an inline `SortableQuestion` helper. It contains:

1. **Header/breadcrumb** (lines 279-302) — back button, breadcrumb, version history sheet
2. **Error banner** (lines 304-309)
3. **Exam metadata card** (lines 313-371) — title, description, duration, sequence inputs
4. **Question list toolbar** (lines 375-388) — section header + "Add Question" buttons (3 types)
5. **Empty state** (lines 390-401) — shown when no questions exist
6. **Question list with drag-and-drop** (lines 402-581) — DndContext + SortableContext wrapping question cards
7. **Per-question content** (lines 420-576) — question text, points, options editor, AI grading config
8. **Action buttons** (lines 585-613) — save draft / publish buttons

All state is local: `formData` (with nested `questions` array), `loading`, `error`. Question mutations (`addQuestion`, `removeQuestion`, `updateQuestion`) and drag-reorder are inline functions.

## Context Design

### `exam-builder-context.tsx`

```typescript
export interface ExamFormData {
  title: string
  description: string
  duration: number
  sequence: number
  status: string
  questions: QuestionData[]
}

export interface ExamBuilderContextValue {
  // State
  formData: ExamFormData
  loading: boolean
  error: string | null

  // Props pass-through
  initialData: ExamBuilderProps['initialData']
  courseId: number
  courseTitle: string

  // Question actions
  addQuestion: (type: QuestionData['question_type']) => void
  removeQuestion: (id: number | string) => void
  updateQuestion: (id: number | string, updates: Partial<QuestionData>) => void
  handleDragEnd: (event: DragEndEvent) => void
  sensors: ReturnType<typeof useSensors>

  // Form actions
  updateField: <K extends keyof ExamFormData>(key: K, value: ExamFormData[K]) => void
  handleSave: (publish: boolean) => Promise<void>
  setError: (error: string | null) => void
}
```

**Key decisions:**
- `sensors` and `handleDragEnd` live in context so `ExamQuestionList` doesn't need to re-create them.
- Question manipulation functions (`addQuestion`, `removeQuestion`, `updateQuestion`) are in context because they are used across multiple sub-components (toolbar adds, card header removes, card body updates).
- The `QuestionData` type stays exported from context for use by sub-components.
- No step-based navigation (unlike lesson-editor/exercise-builder) — the exam builder is a single scrollable page, not a multi-step wizard. Adding step navigation would be over-engineering.

### What NOT to put in context
- DnD library imports (`DndContext`, `SortableContext`, etc.) stay local to `exam-question-list.tsx`.
- `useSortable` stays local to `exam-question-card.tsx`.
- Option-level mutations stay inline in `exam-question-options.tsx` (they call `updateQuestion` from context with the modified options array).

## Component Responsibilities

### `exam-builder.tsx` (~15 lines)
Wraps `ExamBuilderProvider` around `ExamBuilderShell`. Same pattern as lesson-editor and exercise-builder.

```typescript
export function ExamBuilder(props: ExamBuilderProps) {
  return (
    <ExamBuilderProvider {...props}>
      <ExamBuilderShell />
    </ExamBuilderProvider>
  )
}
```

### `exam-builder-header.tsx` (~40 lines)
Extracts lines 279-302. Reads `courseId`, `courseTitle`, `initialData`, `formData` from context.
- Back button (Link to course page)
- Breadcrumb (course title > create/edit)
- Version history sheet (edit mode only)

### `exam-details-card.tsx` (~70 lines)
Extracts lines 313-371. Reads `formData`, `updateField` from context.
- Card with title, description, duration, sequence inputs
- All field changes go through `updateField`

### `exam-questions-toolbar.tsx` (~35 lines)
Extracts lines 375-388. Reads `formData.questions.length`, `addQuestion` from context.
- Section heading with question count
- Three "Add" buttons (multiple choice, true/false, free text)

### `exam-question-list.tsx` (~50 lines)
Extracts lines 390-581 (the outer structure). Reads `formData.questions`, `sensors`, `handleDragEnd` from context.
- Empty state card when no questions
- `DndContext` + `SortableContext` wrapper
- Maps over questions, rendering `<ExamQuestionCard>` for each

### `exam-question-card.tsx` (~60 lines)
Refactored from `SortableQuestion` (lines 72-142) + the `CardContent` (lines 420-430). Reads `updateQuestion`, `removeQuestion` from context.
- `useSortable` hook for drag-and-drop
- Card wrapper with drag handle, type badge, delete button
- Question text textarea + points input
- Renders `<ExamQuestionOptions>` or `<ExamQuestionAIGrading>` based on question type

### `exam-question-options.tsx` (~80 lines)
Extracts lines 445-524. Receives `question: QuestionData` as prop, reads `updateQuestion` from context.
- Correct-answer toggle buttons
- Option text inputs (disabled for true/false)
- Add/remove option buttons (MC only)
- True/false hint text

**Why a separate component:** This is the densest UI block (80 lines of nested mapping with inline handlers). Isolating it makes `exam-question-card.tsx` cleaner and makes the options editor independently testable.

### `exam-question-ai-grading.tsx` (~55 lines)
Extracts lines 528-573. Receives `question: QuestionData` as prop, reads `updateQuestion` from context.
- AI grading notice header
- Grading rubric textarea
- AI grading criteria textarea
- Expected keywords input (comma-separated)

**Why a separate component:** This is a cohesive block for free-text questions only, with its own visual treatment (primary-tinted card). Clean boundary.

### `exam-builder-actions.tsx` (~40 lines)
Extracts lines 585-613. Reads `loading`, `formData`, `handleSave` from context.
- Save draft button (disabled when no title)
- Publish button (disabled when no title or no questions)
- Loading spinner states

## What Stays Inline (Not Extracted)

1. **Error banner** (~6 lines, lines 304-309) — stays in `ExamBuilderShell`. It's too small and too tied to layout positioning to warrant its own file.
2. **Option-level event handlers** — the `onClick` handlers that call `updateQuestion(q.id, { options: newOpts })` stay inline within `exam-question-options.tsx`. These are simple derived calls, not independent logic.

## Props vs Context Access

| Component | Props from parent | Context access |
|-----------|------------------|----------------|
| `ExamBuilderHeader` | none | `courseId`, `courseTitle`, `initialData`, `formData` |
| `ExamDetailsCard` | none | `formData`, `updateField` |
| `ExamQuestionsToolbar` | none | `formData.questions.length`, `addQuestion` |
| `ExamQuestionList` | none | `formData.questions`, `sensors`, `handleDragEnd` |
| `ExamQuestionCard` | `question`, `index` | `updateQuestion`, `removeQuestion` |
| `ExamQuestionOptions` | `question` | `updateQuestion` |
| `ExamQuestionAIGrading` | `question` | `updateQuestion` |
| `ExamBuilderActions` | none | `loading`, `formData`, `handleSave` |

**Rationale for `question` as prop on card/options/ai-grading:** These components render a specific question from the list. Passing the question object as a prop (rather than an index to look up in context) is simpler and avoids re-rendering all cards when any single question changes. The parent (`ExamQuestionList`) already maps over questions and has the data.
