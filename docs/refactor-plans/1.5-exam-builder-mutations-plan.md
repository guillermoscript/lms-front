## Mutations Identified

### Single Composite Mutation: `handleSave` (lines 236-315)

The `handleSave` function performs **all** database writes in a single flow. Unlike lesson/exercise builders which have simple create/update, the exam builder does a multi-step transactional write:

#### Step 1: Create or Update Exam (lines 244-271)
| Property | Value |
|----------|-------|
| **Table** | `exams` |
| **Operation** | INSERT (new) or UPDATE (existing, keyed by `exam_id`) |
| **Fields written** | `course_id`, `title`, `description`, `duration`, `sequence`, `status`, `created_by` |
| **Auth** | `supabase.auth.getUser()` for `created_by` |
| **Tenant filter** | None on write (relies on RLS); should add `tenant_id` to insert and `.eq('tenant_id')` to update |

#### Step 2: Delete All Existing Questions (line 274)
| Property | Value |
|----------|-------|
| **Table** | `exam_questions` |
| **Operation** | DELETE WHERE `exam_id = examId` |
| **Note** | Cascading delete-and-reinsert strategy. Deletes all questions then re-inserts them. This also cascades to `question_options` if FK has ON DELETE CASCADE. |

#### Step 3: Insert Questions (lines 276-293)
| Property | Value |
|----------|-------|
| **Table** | `exam_questions` |
| **Operation** | INSERT (one per question, in a loop) |
| **Fields written** | `exam_id`, `question_text`, `question_type`, `points_possible`, `sequence`, `ai_grading_criteria`, `expected_keywords`, `grading_rubric`, `tenant_id` |
| **Note** | Uses `.select().single()` to get the new `question_id` for options |

#### Step 4: Insert Options (lines 295-304)
| Property | Value |
|----------|-------|
| **Table** | `question_options` |
| **Operation** | INSERT (batch per question, only for `multiple_choice` and `true_false` types) |
| **Fields written** | `question_id`, `option_text`, `is_correct`, `tenant_id` |

#### Post-success Behavior (lines 307-308)
- `router.push(/dashboard/teacher/courses/${courseId})`
- `router.refresh()`

## Security Issues in Current Implementation

1. **No role check** -- only checks `getUser()`, not teacher/admin role
2. **No course ownership verification** -- any authenticated user could write to any course's exams
3. **Client-side `tenantId`** -- passed as prop from server page, but used directly in client writes without server-side re-verification
4. **No input validation** -- title could be empty/whitespace (UI disables button but action doesn't validate)

## Target: `app/actions/teacher/exams.ts`

### Interface

```typescript
export interface ExamQuestionData {
  question_text: string
  question_type: 'multiple_choice' | 'true_false' | 'free_text'
  points_possible: number
  sequence: number
  ai_grading_criteria: string
  expected_keywords: string[]
  grading_rubric?: string
  options: {
    option_text: string
    is_correct: boolean
  }[]
}

export interface ExamFormData {
  title: string
  description: string
  duration: number
  sequence: number
  publish: boolean
  questions: ExamQuestionData[]
}
```

### Server Actions

#### `createExam(courseId: number, data: ExamFormData)`

```
1. requireTeacherOrAdmin()
2. verifyCourseOwnership(ctx, courseId)
3. Validate: title required, questions array valid
4. INSERT into exams (with tenant_id from ctx, created_by from ctx.userId)
5. For each question: INSERT into exam_questions (with tenant_id)
6. For each MC/TF question: batch INSERT into question_options (with tenant_id)
7. revalidatePath(`/dashboard/teacher/courses/${courseId}`)
8. Return { examId }
```

#### `updateExam(courseId: number, examId: number, data: ExamFormData)`

```
1. requireTeacherOrAdmin()
2. verifyCourseOwnership(ctx, courseId)
3. Validate: title required, questions array valid
4. Verify exam belongs to course+tenant
5. UPDATE exams row
6. DELETE all exam_questions WHERE exam_id (cascade handles options)
7. For each question: INSERT into exam_questions (with tenant_id)
8. For each MC/TF question: batch INSERT into question_options (with tenant_id)
9. revalidatePath(`/dashboard/teacher/courses/${courseId}`)
10. revalidatePath(`/dashboard/teacher/courses/${courseId}/exams/${examId}`)
11. Return { examId }
```

### Validation Rules
- `title` must be non-empty after trim
- `duration` must be positive integer (default 60)
- `sequence` must be positive integer (default 1)
- Each question must have non-empty `question_text`
- MC/TF questions must have at least 2 options
- MC/TF questions must have exactly 1 correct option
- `points_possible` must be >= 0

## Changes to exam-builder.tsx

### Remove
- `createClient` import and `supabase` instance (line 7, 148)
- `tenantId` from props interface (line 71) -- no longer needed client-side
- Entire `handleSave` function body (lines 236-315) -- replace with server action calls

### Add
- Import `createExam`, `updateExam` from `@/app/actions/teacher/exams`
- New `handleSave` that calls the appropriate server action:

```typescript
const handleSave = async (publish: boolean) => {
  setLoading(true)
  setError(null)

  const actionData: ExamFormData = {
    title: formData.title,
    description: formData.description,
    duration: formData.duration,
    sequence: formData.sequence,
    publish,
    questions: formData.questions.map(q => ({
      question_text: q.question_text,
      question_type: q.question_type,
      points_possible: q.points_possible,
      sequence: q.sequence,
      ai_grading_criteria: q.ai_grading_criteria,
      expected_keywords: q.expected_keywords,
      grading_rubric: q.grading_rubric,
      options: q.options.map(o => ({
        option_text: o.option_text,
        is_correct: o.is_correct,
      })),
    })),
  }

  const result = initialData?.exam_id
    ? await updateExam(courseId, initialData.exam_id, actionData)
    : await createExam(courseId, actionData)

  if (result.success) {
    router.push(`/dashboard/teacher/courses/${courseId}`)
    router.refresh()
  } else {
    setError(result.error)
  }

  setLoading(false)
}
```

### Update Pages
- Remove `tenantId` prop from ExamBuilder in both pages (no longer needed)
- The `tenantId` is resolved server-side inside the action via `requireTeacherOrAdmin()`

## Complexity Notes

This is more complex than lessons (1 table) or exercises (1 table) because:
- **3 tables** are mutated: `exams`, `exam_questions`, `question_options`
- **Nested loop writes**: questions are inserted one-by-one (need returned `question_id`), then options are batch-inserted per question
- **Delete-and-reinsert strategy** for questions on update -- simpler than diffing but means the action must handle the full set atomically
- If any step fails mid-loop, partial data may exist. Consider wrapping in a Supabase RPC or at minimum documenting this limitation.
