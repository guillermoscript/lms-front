# 0.1 — Add Suspense Boundaries to Dashboard Layouts

## Rule Reference

**async-suspense-boundaries** (HIGH impact): Instead of awaiting data in async components before returning JSX, use Suspense boundaries to show the wrapper UI faster while data loads.

## Current State

### Layouts with async data fetching that blocks rendering:

1. **`app/[locale]/dashboard/layout.tsx`** — Awaits `getUserRole()`, `createClient()`, and `supabase.auth.getUser()` before rendering sidebar + header. The entire shell (sidebar, header, children) is blocked until auth resolves.

2. **`app/[locale]/platform/layout.tsx`** — Awaits `createClient()`, `supabase.auth.getUser()`, `isSuperAdmin()`, and a `platform_payment_requests` count query. The entire shell is blocked by 4 sequential async operations.

3. **`app/[locale]/dashboard/student/courses/[courseId]/layout.tsx`** — Awaits `createClient()`, `getCurrentTenantId()`, and a `course_ai_tutors` query before rendering children.

### Pages — all are fully async, blocking entire page render:

Every page file examined (`student/page.tsx`, `student/progress/page.tsx`, `student/certificates/page.tsx`, `teacher/page.tsx`, `admin/page.tsx`, `admin/products/page.tsx`, `admin/enrollments/page.tsx`, `platform/page.tsx`, `platform/tenants/page.tsx`) follows the same pattern: all data is fetched at the top of the component, blocking the entire page JSX from rendering.

### Existing loading.tsx files:

- `app/[locale]/dashboard/loading.tsx` — Generic dashboard skeleton
- `app/[locale]/dashboard/student/loading.tsx` — Student-specific skeleton
- `app/[locale]/dashboard/teacher/loading.tsx` — Teacher-specific skeleton
- `app/[locale]/dashboard/admin/loading.tsx` — Admin-specific skeleton
- `app/[locale]/platform/loading.tsx` — Platform skeleton

These provide route-level Suspense via Next.js automatic `loading.tsx` boundaries. They show while the page component's async data loads. This is already good coverage for page-level transitions.

## Analysis & Recommendation

### What NOT to change

1. **Page-level data fetching**: The existing `loading.tsx` files already provide Suspense boundaries for each page. When you navigate to `/dashboard/student`, Next.js wraps the page in `<Suspense fallback={<Loading />}>` automatically. Adding manual Suspense inside these pages would create redundant loading states and potential layout shift. The current pattern is correct for pages.

2. **Auth checks in layouts**: The `getUserRole()` and `supabase.auth.getUser()` calls in `dashboard/layout.tsx` are needed BEFORE rendering because:
   - `AppSidebar` needs `userRole` to show correct navigation items
   - `UserNav` needs `user` data
   - These are critical layout decisions — the rule explicitly says "When NOT to use: Critical data needed for layout decisions"

3. **Platform layout auth guard**: `platform/layout.tsx` must await `isSuperAdmin()` before rendering because it redirects unauthorized users. This is a security boundary, not a data-display concern.

4. **Student course layout**: The `course_ai_tutors` query determines whether the Aristotle AI panel renders at all. This is a layout decision.

### What SHOULD change

The one area where Suspense can genuinely help is the **platform layout's pending billing count** query. This query is NOT a layout decision — it only feeds a badge number on the sidebar. The sidebar can render immediately with the badge hidden/showing 0, then stream in the count.

Similarly, in the **dashboard layout**, the `GamificationHeaderCard` is already a client component with its own loading state — no change needed there. But the `UserNav` component receives the full `user` object. We can render the header shell immediately and stream in the user-dependent parts.

However, there is a significant constraint: **the sidebar requires `userRole`**, and `getUserRole()` also calls `getUser()` internally. Since the sidebar is the primary layout element and needs the role, we cannot avoid the auth fetch. The `user` object for `UserNav` comes from the same `getUser()` call, so there is no additional blocking cost.

### Verdict

**The current architecture is already well-optimized for Suspense.** The key findings:

1. **Layouts await auth data that IS needed for layout decisions** (sidebar navigation, auth guards). The Suspense rule explicitly excludes this case.
2. **Pages ARE wrapped in Suspense** via Next.js `loading.tsx` files with role-appropriate skeletons.
3. **Client components handle their own loading** (GamificationHeaderCard, MiniLeaderboard).

The only actionable improvement is extracting the pending billing count fetch in the platform layout into a streamed component.

## Changes

### Change 1: Platform layout — Stream pending billing badge

**File:** `app/[locale]/platform/layout.tsx`

**Rule:** async-suspense-boundaries — "Sidebar, Header, and Footer render immediately. Only DataDisplay waits for data."

**Rationale:** The `platform_payment_requests` count query blocks the entire platform shell from rendering. The badge count is cosmetic — the sidebar renders fine without it. Extract into a separate async component wrapped in Suspense.

**Before:**
```tsx
export default async function PlatformLayout({
  children,
  params,
}: {
  children: React.ReactNode
  params: Promise<{ locale: string }>
}) {
  const { locale } = await params
  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()

  if (!user || !(await isSuperAdmin())) {
    redirect(`/${locale}/auth/login`)
  }

  // Get pending billing count for badge
  const adminClient = createAdminClient()
  const { count: pendingCount } = await adminClient
    .from('platform_payment_requests')
    .select('*', { count: 'exact', head: true })
    .eq('status', 'pending')

  return (
    <SidebarProvider>
      <PlatformSidebar pendingBillingCount={pendingCount ?? 0} />
      {/* ... rest */}
    </SidebarProvider>
  )
}
```

**After:**
```tsx
import { Suspense } from 'react'

// Async component that fetches and passes the count
async function PlatformSidebarWithCount() {
  const adminClient = createAdminClient()
  const { count: pendingCount } = await adminClient
    .from('platform_payment_requests')
    .select('*', { count: 'exact', head: true })
    .eq('status', 'pending')

  return <PlatformSidebar pendingBillingCount={pendingCount ?? 0} />
}

export default async function PlatformLayout({
  children,
  params,
}: {
  children: React.ReactNode
  params: Promise<{ locale: string }>
}) {
  const { locale } = await params
  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()

  if (!user || !(await isSuperAdmin())) {
    redirect(`/${locale}/auth/login`)
  }

  return (
    <SidebarProvider>
      <Suspense fallback={<PlatformSidebar pendingBillingCount={0} />}>
        <PlatformSidebarWithCount />
      </Suspense>
      <SidebarInset>
        <header className="flex h-16 shrink-0 items-center gap-2 border-b px-4">
          <SidebarTrigger className="-ml-1" />
          <Separator orientation="vertical" className="mr-2 h-4" />
          <div className="flex items-center gap-2">
            <span className="text-sm font-medium text-muted-foreground">Platform Admin</span>
          </div>
          <div className="ml-auto flex items-center gap-4">
            <ModeToggle />
            <UserNav user={user} />
          </div>
        </header>
        <div className="flex flex-1 flex-col">
          {children}
        </div>
      </SidebarInset>
    </SidebarProvider>
  )
}
```

**Impact:** The platform layout shell (header, content area) renders immediately after auth check. The sidebar renders instantly with `pendingBillingCount={0}` as fallback, then the real count streams in. The DB query no longer blocks the shell.
