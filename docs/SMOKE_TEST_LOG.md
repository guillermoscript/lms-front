## Critical Flow Test — 2026-05-16

**Method:** Chrome extension. End-to-end multi-step flows (not just page loads),
exercised as the real personas. LMS dev server on **port 3001**.

**Result:** 2 of 3 critical flows pass. 1 broken flow + 1 duplicate-write bug found.

### Flow 1 — Teacher: create course → lesson → publish ✅ PASS

Account: `creator@codeacademy.com` (Code Academy admin). Steps verified:
1. Create Course form → "Smoke Test Flow Course" created (course `10003`).
2. Add lesson → title + summary saved → block-editor content added.
3. Lesson published; course Status set to `published` via Settings.

Result: course is published and visible to students in the public catalog.

⚠️ **Bug — duplicate lesson on repeated Save Draft.** The lesson editor at
`/courses/{id}/lessons/new` does **not** redirect to an edit URL after the first
"Save Draft". A second "Save Draft" click therefore created a **second
identical lesson** (course ended up with 2 × "Lesson 1 - Getting Started", both
display-order 1). Fix: after first save, replace the route with
`/lessons/{newId}` (the edit page) so subsequent saves update instead of insert.

### Flow 2 — Student: enrollment ⚠️ PARTIAL / BROKEN

Account: `alice@student.com` (Code Academy student, no active subscription).

- Dashboard browse (`/dashboard/student/browse`) correctly **gates** enrollment:
  with no subscription the course card shows "🔒 Subscribe to Access". ✅
- Public course page (`/courses/10003`) shows the same course as **"Free"** with
  an "Enroll for Free" button — **inconsistent** with the dashboard gate.
- Clicking "Enroll for Free" → `/checkout?courseId=10003` → "Enroll for Free":
  the POST returns **HTTP 500**, with **no error shown to the user** (the button
  silently reverts). Enrollment fails.

🔴 **Bug — free enrollment 500.** Root cause in
`app/[locale]/(public)/checkout/actions.ts` → `enrollFree()`. For a course with
no linked `product_courses` row it tries to `INSERT` into `products`,
`product_courses`, and `enrollments` **without `tenant_id`** — these are
tenant-scoped RLS tables, and a student session has no INSERT grant on
`products`. So free enrollment only works for courses that already have a
product attached; brand-new product-less courses cannot be enrolled in.
Fixes: (a) set `tenant_id` on all three inserts, (b) create the free product
with an admin/service client, (c) surface the error to the UI instead of
silently reverting.

✅ **FIXED (2026-05-16).** `app/[locale]/(public)/checkout/actions.ts`:
- `enrollFree()` rewritten — validates the course (tenant + `published`),
  creates the per-course free `products` + `product_courses` rows via
  `createAdminClient()` with explicit `tenant_id`, then enrolls through the
  `enroll_user` RPC (which derives `tenant_id` and upserts on
  `(user_id, course_id)` — idempotent).
- `enrollUser()` — `transactions` inserts (course + plan branches) now set
  `tenant_id` (was defaulting to the default tenant → broke paid checkout for
  every other tenant too).
Verified: Alice enrolled in course 10003 on Code Academy; `enrollment`,
`products`, `product_courses` rows all written with `tenant_id =
…0002` (not the default `…0001`). Re-enrollment is idempotent.

### Flow 3 — Student: lesson → exam → AI grading → certificate ✅ PASS

Account: `alice@student.com`, course "Quick Test Course" (already enrolled).

1. Course view + curriculum render; lesson viewer shows content, comments,
   per-lesson completion sidebar. ✅
2. Exam page (`/courses/9999/exams`) → "Quick Test Exam" completed, 100%. ✅
3. Result page shows **"AI Evaluated"** banner — AI grading pipeline produced
   scores/feedback. ✅
4. "Certificate Earned!" → certificate → public verification page

⚠️ **Progress mismatch.** The course overview page shows **0% / 0/2 lessons**,
but the in-lesson sidebar shows **2/2 / 100%** for the same course/student.
Two different progress sources disagree.

ℹ️ **i18n gaps found during the flows:**

ℹ️ Certificate verify page renders the student name in dark green on a dark
background — very low contrast (a11y).

### Findings & observations

1. **⚠️ LMS dev server was not running on port 3000.** Port 3000 was occupied by
   a different project ("Rentaflow", a vacation-rental PMS). The LMS was started
   on **port 3001** for this run (`npm run dev -- -p 3001` with
   `NEXT_PUBLIC_PLATFORM_DOMAIN=lvh.me:3001`). Use port 3000 with the default
   platform domain for normal runs / for the Playwright suite.

2. **ℹ️ `owner@e2etest.com` is an ADMIN, not a teacher.** `tests/playwright/utils/constants.ts`
   labels this account `teacher`, but it is the Default School owner and resolves
   to the `admin` role — it lands on `/dashboard/admin`. There is no dedicated
   teacher-role test account. The `/dashboard/teacher/*` routes are still
   reachable and render correctly with this account.

3. **⚠️ Dev server exited once mid-run** (clean exit, code 0) and had to be
   restarted. No crash in the logs — possibly the background-task runner reaping
   a long-lived process. Worth watching if it recurs.

4. **ℹ️ Repeated server-log advisory:** `Using the user object as returned from
   supabase.auth.getSession() ... could be insecure! ... Use supabase.auth.getUser()
   instead.` Appears on multiple dashboard routes. This is a known/intentional
   trade-off (`getSession()` adopted for auth performance — see project memory),
   but the advisory is still emitted on every affected request.

5. **ℹ️ Next.js dev warning:** `Cross origin request detected from
   code-academy.lvh.me to /_next/* resource` — recommends configuring
   `allowedDevOrigins` in `next.config`. Dev-only; harmless but easy to silence.

6. **ℹ️ Several pages render an empty `<h1>`** (teacher dashboard / courses /
   templates / api-tokens; admin dashboard / landing-page). Content renders fine
   via other heading levels — minor a11y/SEO observation, not a failure.

7. **ℹ️ Community is feature-gated.** Student, teacher, and admin community pages
   all show "Upgrade to Starter" because both test schools are on the Free plan.
   Expected behavior.

8. **Console-error capture:** zero console errors for teacher and admin runs
   (tracking active). The student run was completed before console tracking was
   enabled, so student console output was not captured this run.
