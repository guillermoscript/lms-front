## Findings considered and rejected (so nobody re-audits them)

From the audit's verify pass — confirmed NOT worth a plan:

- **Public certificate-verify endpoint "leaks" name/course** (`certificates/verify/[code]`): by design — documented public credential feature (API_REFERENCE.md, certificate_shares table). Not a vuln.
- **`user_roles` RLS "recursion"** in certificate/invoice routes: false-positive — queries filter `user_id = auth.uid()` (own roles); the recursion bug was fixed by migration `20260207000001`.
- **LemonSqueezy webhook secret fallback undefined → silent fail**: false-positive — the provider factory (`lib/payments/index.ts:40-50`) throws on missing secret before instantiation; the fallback is unreachable.
- **Webhook idempotency → double-extension / wrong end_date**: false-positive — `extend_subscription_period` sets `end_date` from the event's `periodEnd` (not `now()`), so re-running is idempotent (dispatcher comment confirms). (Test gap is real and covered by plan 006's follow-ups.)
- **Header-spoof RLS bypass (no tenant_id claim)**: by design / already closed — migration `20260612170000_harden_get_tenant_id` makes `get_tenant_id()` fail closed (NULL) for tokens lacking the claim. API-layer defense-in-depth is a nice-to-have, not a current hole.
- **Stripe SDK major bumps** (20→22, JS 8→9): not worth doing now — no concrete need; high churn risk on live payments. Revisit on demand.
- **122 migrations flat/chronological**: by design — standard Supabase migration layout; reorganizing risks history confusion for marginal benefit.
- **Most perf findings** (PERF-03/04/05/06/09): false-positive or low-signal on re-read (RSC already used, lists already bounded, counts are cheap). Three genuine perf quick-wins (N+1 lesson reorder, `select(*)` over-fetch, sequential admin-dashboard fetch) were noted but not selected into this batch — available for a future `perf` run.
